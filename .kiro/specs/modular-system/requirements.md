# Requirements Document

## Introduction

Модульная система для платформы MasterPlan, позволяющая расширять функционал через подключаемые приложения (модули). Пользователи смогут включать/отключать модули, а в будущем — покупать их в каталоге приложений.

## Glossary

- **Module**: Самостоятельный пакет функционала, который можно подключить к основной системе
- **Module_Registry**: Реестр всех доступных модулей в системе
- **User_Module**: Связь между пользователем и подключённым модулем
- **Hook_Point**: Точка расширения в основной системе, куда модуль может добавить свой функционал
- **Module_Manifest**: JSON-файл с метаданными модуля (название, версия, зависимости)
- **Core_System**: Базовый функционал платформы (календарь, клиенты, записи)

## Requirements

### Requirement 1: Регистрация и обнаружение модулей

**User Story:** As a система, I want автоматически обнаруживать и регистрировать модули, so that новые модули становятся доступны без ручной настройки.

#### Acceptance Criteria

1. WHEN система запускается, THE Module_Registry SHALL сканировать директорию `app/Modules` и регистрировать все найденные модули
2. WHEN модуль содержит валидный `module.json`, THE Module_Registry SHALL добавить его в список доступных модулей
3. IF модуль не содержит обязательных полей в манифесте, THEN THE Module_Registry SHALL пропустить модуль и записать ошибку в лог
4. THE Module_Manifest SHALL содержать поля: slug, name, description, version, author, dependencies, hooks, routes, permissions

### Requirement 2: Управление модулями пользователем

**User Story:** As a мастер, I want включать и отключать модули в настройках, so that я использую только нужный мне функционал.

#### Acceptance Criteria

1. WHEN пользователь открывает страницу "Приложения", THE Core_System SHALL отобразить список всех доступных модулей с их статусом (включён/выключен)
2. WHEN пользователь нажимает "Включить" на модуле, THE Core_System SHALL создать запись в User_Module и активировать функционал модуля
3. WHEN пользователь нажимает "Выключить" на модуле, THE Core_System SHALL деактивировать модуль, но сохранить данные пользователя
4. IF модуль имеет зависимости от других модулей, THEN THE Core_System SHALL предупредить пользователя и предложить включить зависимости
5. WHEN модуль включается, THE Core_System SHALL выполнить миграции модуля, если они ещё не выполнены

### Requirement 3: Система хуков (точек расширения)

**User Story:** As a модуль, I want добавлять свой функционал в определённые места интерфейса, so that пользователь видит функционал модуля в контексте основной системы.

#### Acceptance Criteria

1. THE Core_System SHALL предоставить следующие Hook_Points:
   - `sidebar.menu` — добавление пунктов в боковое меню
   - `dashboard.widgets` — добавление виджетов на дашборд
   - `client.card.tabs` — добавление вкладок в карточку клиента
   - `appointment.form.fields` — добавление полей в форму записи
   - `settings.sections` — добавление разделов в настройки
   - `public.page.sections` — добавление секций на публичную страницу
2. WHEN модуль регистрирует хук, THE Core_System SHALL вызывать callback модуля при рендеринге соответствующей точки
3. WHEN модуль отключён, THE Core_System SHALL НЕ вызывать его хуки

### Requirement 4: Маршрутизация модулей

**User Story:** As a модуль, I want иметь собственные маршруты API и страницы, so that я могу предоставлять полноценный функционал.

#### Acceptance Criteria

1. WHEN модуль активен, THE Core_System SHALL загружать маршруты модуля из `Modules/{ModuleName}/Routes/`
2. THE Core_System SHALL автоматически добавлять префикс `/api/modules/{module-slug}` к API-маршрутам модуля
3. THE Core_System SHALL автоматически добавлять префикс `/app/modules/{module-slug}` к web-маршрутам модуля
4. IF модуль отключён, THEN THE Core_System SHALL возвращать 404 для всех маршрутов модуля

### Requirement 5: Права доступа модулей

**User Story:** As a система, I want контролировать доступ к функционалу модулей, so that пользователи видят только доступный им функционал.

#### Acceptance Criteria

1. WHEN пользователь пытается использовать функционал модуля, THE Core_System SHALL проверить, что модуль включён для этого пользователя
2. IF модуль требует определённый тарифный план, THEN THE Core_System SHALL проверить подписку пользователя
3. WHEN модуль определяет собственные permissions, THE Core_System SHALL учитывать их при проверке доступа
4. THE Core_System SHALL предоставить middleware `module.active:{slug}` для защиты маршрутов

### Requirement 6: Хранение данных модулей

**User Story:** As a модуль, I want иметь собственные таблицы и хранить данные, so that я могу сохранять состояние и настройки.

#### Acceptance Criteria

1. THE Core_System SHALL позволять модулям создавать собственные миграции в `Modules/{ModuleName}/Database/Migrations/`
2. WHEN модуль включается впервые, THE Core_System SHALL выполнить все миграции модуля
3. THE Core_System SHALL предоставить таблицу `module_settings` для хранения настроек модулей в формате JSON
4. WHEN модуль отключается, THE Core_System SHALL НЕ удалять данные модуля (только деактивировать)

### Requirement 7: Frontend-интеграция модулей

**User Story:** As a модуль, I want иметь собственные React-компоненты и страницы, so that я могу предоставлять пользовательский интерфейс.

#### Acceptance Criteria

1. THE Core_System SHALL загружать React-компоненты модулей из `resources/js/modules/{module-slug}/`
2. WHEN модуль регистрирует страницы, THE Core_System SHALL добавлять их в роутинг Inertia.js
3. THE Core_System SHALL предоставить React-контекст `useModules()` для проверки активности модулей
4. WHEN модуль отключён, THE Core_System SHALL НЕ рендерить его компоненты в хуках

### Requirement 8: Каталог приложений (базовая версия)

**User Story:** As a мастер, I want видеть каталог доступных приложений с описаниями, so that я могу выбрать нужные мне модули.

#### Acceptance Criteria

1. THE Core_System SHALL отображать страницу "Каталог приложений" со списком всех модулей
2. WHEN пользователь открывает каталог, THE Core_System SHALL показать для каждого модуля: название, описание, иконку, категорию, статус (включён/выключен/платный)
3. THE Core_System SHALL группировать модули по категориям (Финансы, Маркетинг, Коммуникации, Аналитика)
4. WHEN пользователь нажимает на модуль, THE Core_System SHALL показать детальную страницу с полным описанием и скриншотами

### Requirement 9: Покупка модулей (полный функционал)

**User Story:** As a мастер, I want покупать платные модули через удобный интерфейс, so that я могу расширить функционал своей CRM.

#### Acceptance Criteria

1. THE Module_Manifest SHALL поддерживать поле `pricing` с типами: free, subscription, one-time
2. THE Core_System SHALL хранить информацию о покупках модулей в таблице `module_purchases`
3. IF модуль платный и не куплен, THEN THE Core_System SHALL показывать кнопку "Купить" с ценой
4. WHEN пользователь нажимает "Купить", THE Core_System SHALL показать модальное окно подтверждения покупки (shadcn Dialog)
5. WHEN пользователь подтверждает покупку, THE Core_System SHALL создать платёж через ЮKassa
6. WHEN платёж успешен, THE Core_System SHALL автоматически активировать модуль и показать уведомление (shadcn Toast)
7. IF модуль имеет тип subscription, THEN THE Core_System SHALL показывать дату следующего списания и кнопку отмены подписки
8. THE Core_System SHALL отображать историю покупок модулей в разделе "Мои приложения"
9. WHEN подписка на модуль истекает, THE Core_System SHALL отключить модуль и уведомить пользователя

### Requirement 10: Статистика модулей в админке

**User Story:** As a администратор платформы, I want видеть статистику использования модулей, so that я могу анализировать популярность и доходность.

#### Acceptance Criteria

1. THE Admin_Panel SHALL отображать страницу "Статистика модулей" с общими метриками
2. WHEN администратор открывает статистику, THE Admin_Panel SHALL показать:
   - Количество установок каждого модуля
   - Количество активных пользователей модуля
   - Доход от каждого модуля (для платных)
   - График установок по времени
3. THE Admin_Panel SHALL позволять фильтровать статистику по периоду (день, неделя, месяц, год)
4. WHEN администратор выбирает конкретный модуль, THE Admin_Panel SHALL показать детальную статистику:
   - Список пользователей с модулем
   - История покупок
   - Отзывы/рейтинг (если есть)
5. THE Admin_Panel SHALL отображать топ-5 популярных модулей на дашборде

### Requirement 11: Управление модулями в админке

**User Story:** As a администратор платформы, I want управлять модулями и их настройками, so that я могу контролировать доступность и цены.

#### Acceptance Criteria

1. THE Admin_Panel SHALL отображать список всех модулей с возможностью редактирования
2. WHEN администратор редактирует модуль, THE Admin_Panel SHALL позволять изменить:
   - Цену модуля
   - Тип монетизации (free/subscription/one-time)
   - Минимальный тарифный план
   - Статус (активен/отключён для всех)
   - Описание и скриншоты
3. THE Admin_Panel SHALL позволять временно отключить модуль для всех пользователей
4. WHEN модуль отключается глобально, THE Core_System SHALL уведомить всех пользователей с этим модулем
5. THE Admin_Panel SHALL позволять предоставить модуль бесплатно конкретному пользователю
6. THE Admin_Panel SHALL показывать логи ошибок модулей для отладки

### Requirement 12: События и интеграция между модулями

**User Story:** As a модуль, I want реагировать на события других модулей и ядра, so that я могу интегрироваться с другим функционалом.

#### Acceptance Criteria

1. THE Core_System SHALL публиковать события: `appointment.created`, `appointment.completed`, `client.created`, `payment.received`
2. WHEN модуль подписывается на событие, THE Core_System SHALL вызывать обработчик модуля при возникновении события
3. THE Core_System SHALL позволять модулям публиковать собственные события с префиксом `module.{slug}.{event}`
4. IF модуль-источник события отключён, THEN THE Core_System SHALL НЕ публиковать его события
