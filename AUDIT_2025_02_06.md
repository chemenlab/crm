# I. АРХИТЕКТУРА БЭКЕНДА (Laravel)

## 1. Общая архитектура и организация кода — 6/10

### Плюсы

- Разумная структура каталогов с разделением на контексты: `Controllers\App`, `Controllers\Admin`, `Controllers\Auth`, `Controllers\Public`, `Controllers\Webhooks`
- Модульная система (`app/Modules/`) с собственными контроллерами, моделями, миграциями и маршрутами
- Сервисный слой (`app/Services/`): `PaymentService`, `SubscriptionService`, `UsageLimitService`, `TimeSlotGenerator`

### Проблемы

**1.1. Дублирование контроллеров настроек**

Обнаружено дублирование файлов:
- `app/Settings/ProfileController.php`
- `app/Http/Controllers/App/Settings/ProfileController.php`

Аналогично для `CustomFieldsController.php`, `FieldsController.php`, `NotificationSettingsController.php`. Директория `app/Settings/` — артефакт рефакторинга.

**1.2. Дублирование NotificationService**

- `app/Services/NotificationService.php` (строки 1-65) — заглушки с TODO
- `app/Services/Notifications/NotificationService.php` — действующая реализация

**1.3. Дублирование обработки исключений**

Обработка определена и в `bootstrap/app.php` (строки 34-122), и в `app/Exceptions/Handler.php` (строки 44-158). В Laravel 12 правильный подход — `bootstrap/app.php`, а `Handler.php` — артефакт.

**1.4. Регистрация событий в AppServiceProvider**

`app/Providers/AppServiceProvider.php` (строки 39-52): события регистрируются через `Event::listen()` вместо декларативного `EventServiceProvider` или атрибутов PHP 8.1+.

## 2. API и структура маршрутов — 5/10

### Проблемы

**2.1. Отсутствие `routes/api.php`**

Все API-маршруты в `routes/web.php` (строки 707-736) с префиксом `/api`, проходят через web middleware stack (сессии, CSRF, cookies).

**2.2. Монолитный файл маршрутов**

`routes/web.php` — 737 строк. Следует разделить на `routes/admin.php`, `routes/api.php`, `routes/public.php`, `routes/app.php`.

**2.3. Несогласованное использование HTTP-методов**

- Строка 112: `Route::patch('/services/{service}')` — PATCH
- Строка 186: `Route::put('/settings/profile')` — PUT

Нет единой конвенции.

**2.4. Дублирование regex-ограничений**

Строки 370-410: regex `'^(?!leads$|reviews$)[a-z0-9-]+$'` повторяется 11 раз. Нарушение DRY.

**2.5. `env()` в маршрутах**

`routes/web.php`, строка 414: `$adminPath = env('ADMIN_PANEL_PATH', 'admin')` — не работает при `php artisan config:cache`.

**2.6. Тестовый маршрут в production**

Строки 98-101: `/test/dashboard-components` не должен быть в production.

## 3. Модели и схема БД — 7/10

### Плюсы

- Хорошее использование Eloquent-отношений, `$casts`, скоупов, аксессоров

### Проблемы

**3.1. God Object — модель User**

`app/Models/User.php` — 504 строки, 35+ отношений, бизнес-логика подписок, OAuth, 2FA, каскадное удаление. Нарушает SRP. Рекомендуется:
- Выделить трейты: `HasSubscription`, `HasTwoFactor`, `HasOAuth`
- Перенести логику удаления в `UserDeletionService`

**3.2. Каскадное удаление без транзакций**

`User.php`, строки 98-124: удаление 24+ связанных моделей без `DB::transaction()`. При исключении — неконсистентное состояние данных.

**3.3. Перегруженная таблица `users`**

30+ fillable-полей, включая `site_title`, `site_description`, `theme_color`, `site_gradient_from` — должны быть в отдельной таблице `user_profiles`.

**3.4. Отсутствие Enum-типов для статусов**

Статусы записей (`scheduled`, `confirmed`, `completed`, `cancelled`, `no_show`) прописаны строками в 3+ файлах. В модуле Leads Enums используются, в ядре — нет.

**3.5. Связывание Module через slug вместо id**

`Module` (строки 52-55) использует `module_slug` как foreign key. Slug может измениться.

## 4. Middleware и авторизация — 7/10

### Плюсы

- Отдельный admin guard с IP-whitelist
- RBAC для администраторов
- Middleware для проверки лимитов подписки

### Проблемы

**4.1. [КРИТИЧНО] Утечка хеша пароля на фронтенд**

`HandleInertiaRequests.php`, строка 132:
```php
'password' => $request->user()->password,
```
Хеш пароля передается в браузер. Правильно: `'has_password' => (bool) $request->user()->password`.

**4.2. N+1 на каждый запрос**

`HandleInertiaRequests.php`, строки 46-52: загружаются ВСЕ незакрытые тикеты, фильтруются в PHP. Происходит на КАЖДЫЙ запрос.

**4.3. Избыточные данные в shared props**

Строки 86-111: 3 дополнительных запроса к БД (`clients()->count()`, `services()->count()`, `appointments()->count()`) на каждый запрос, хотя нужны только на дашборде.

**4.4. Нет Rate Limiting на публичных маршрутах**

Публичные маршруты бронирования и вебхуки без rate limiting.

### Критические проблемы

**6.1. [КРИТИЧНО] Утечка хеша пароля**

`HandleInertiaRequests.php:132` — хеш пароля в браузер.

**6.2. [КРИТИЧНО] Обход проверки webhook-подписи**

`PaymentService.php`, строки 375-386:
```php
if (!$webhookSecret) {
    return true; // Пропускаем проверку
```

`AppointmentController.php`, строки 411-428:

```php
      ->orWhereNull('user_id')
```

Любой пользователь может удалить записи без владельца. Тот же паттерн в `ClientController.php` (строки 331-337), `ServiceController.php` (строки 170-176).

**6.4. Нет rate limiting на аутентификации**

**6.5. Отсутствие авторизации в `store()`**

`AppointmentController.php:136`: метод `store()` не использует Policy, в отличие от `edit()` и `update()`.
